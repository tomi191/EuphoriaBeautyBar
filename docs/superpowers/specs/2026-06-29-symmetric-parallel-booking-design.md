## 1. Проблем (диагностициран с prod данни)

Снежана има на 30.06.2026 (вторник, работи 09:00–19:00):
- `14:00–16:15` Балаяж (къса коса) — active=50, processing=40 (валиден host)
- `17:30–19:15` **„Боядисване"** — старата единична услуга, `active=0, processing=0, bookableOnline=false`

Иска да добави **второ боядисване в 17:00** (по-рано от съществуващия). Системата отказва.

**Два наслоени root cause-а (потвърдени чрез `scripts/diag-snezhana-3006*.ts`):**

1. **Часът от 17:30 е със стара услуга без престой** (`active=0/proc=0`). `windowFor` връща `null` за processing ≤ 10 → не генерира паралелен прозорец → нищо не може да се сложи паралелно с нея.
2. **Моделът е еднопосочен.** `windowFor(start, active, proc)` винаги връща прозорец *след* `start`. Затова паралел се вмъква само в престоя на вече записан **по-ранен** host. Снежана иска обратното — по-**ранен** час (17:00) преди записан (17:30). Прозорецът на 17:30 е след 17:30 и не покрива 17:00.

**Допълнителни находки (от /staff през Playwright):**
- Редакцията на услуга в /staff (`my-services.tsx` → `EditSheet`) има само цена/продължителност/макс/валута/„от" — **няма поле за престой**. Затова Снежана не може да го настройва; то е само в admin (`service-item-form.tsx`).
- Старата „Боядисване" (90 мин, 0/0) е активна и онлайн при Снежана — капанът.
- Хаотичен ред на услугите + дублирани стари услуги (отделна задача — ценоразпис).

## 2. Желано поведение (потвърдено от потребителя)

Реалната работа на Снежана: намазва клиент А (17:00), щом свърши намазването започва да намазва клиент Б (17:30), докато А „седи" с боята. Намазванията се редят едно след друго; престоите се застъпват.

**Правило (симетрично):** нов час се вписва паралелно, когато:
1. **Намазването (active фаза) не се застъпва** с намазването на никой друг неотменен час за ресурса, с поне `PARALLEL_SAFETY_MIN = 5` минути между края на едното намазване и началото на другото. ← твърда гаранция (Снежана има едни ръце)
2. **Намазването попада в нечий престой** — *или* новото намазване ляга в престоя на застъпен час, *или* намазването на застъпен час ляга в новия престой. Това прави правилото симетрично (редът на запис няма значение) и отрязва намазване върху чуждо измиване.
3. Намазването е в работния ден (престоят може да прелее след работно време — както сега).

Измиването/финалната обработка **не** се заключва твърдо — Снежана я управлява с таймера (както досега). MVP без изкуствен таван „1 паралелен" — ограничението идва естествено от правило 1.

Промяната важи **само за екипния календар** (/staff). Публичната онлайн форма (`/zapazi-chas`) остава block-based (по-предпазлива — клиент да не трупа застъпени дълги часове).

## 3. Архитектура — част A (симетричен паралел)

### 3.1 Pure ядро (`src/lib/booking/parallel-window.ts`)

Запазваме `windowFor` (престой-прозорец на час) и `PARALLEL_SAFETY_MIN`. Добавяме:

```ts
// Чистата активна (намазване) фаза на час: [startMs, startMs + activeMin*60000].
export function activePhase(startMs, activeMin): { start: number; end: number }

// Симетрична проверка за един съществуващ час спрямо кандидат.
export function pairFits(
  candStart, candActive, candProc,
  otherStart, otherActive, otherProc,
): boolean
//  true ако:
//   (a) намазванията НЕ се застъпват с поне SAFETY мин помежду им:
//       candActive.start >= otherActive.end + SAFETY*60000  ИЛИ
//       otherActive.start >= candActive.end + SAFETY*60000
//   (b) candActive ⊆ windowFor(other...)  ИЛИ  otherActive ⊆ windowFor(cand...)
//       (windowFor вече включва ±SAFETY около престоя)
```

Бележки:
- `windowFor` връща `null` ако processing ≤ 2·SAFETY → за такива услуги няма престой; правило 2(b) се проваля → не паралел (плътен зает блок). Това коректно изключва услуги без престой (стрижки, старата „Боядисване").
- Всичко pure (без DB) — за unit тестове.

### 3.2 DB слой (`src/lib/booking/parallel.ts`)

`fitsParallelWindow(...)` → нова `fitsParallelSlot(resourceId, start, activeMin, processingMin, excludeId?)`:
- Зарежда неотменените часове за деня (без excludeId).
- За **всеки застъпен** съществуващ час проверява `pairFits(...)`. Връща `true` само ако кандидатът пасва с **всеки** застъпен час (никое намазване не се блъска) И поне един осигурява правило 2(b).
- Ако кандидатът не застъпва никой → не е „паралел" случай (връща се отделно като свободен в slot computation).

### 3.3 Slot computation (`compute-slots.ts` + `slots.ts`)

`computeDaySlots` получава вместо абстрактните `wins` — масив от съществуващите часове с техните фази:
```ts
interface BusyBooking { start: number; end: number; activeMin: number; processingMin: number }
```
За слот `t` (нов час с `activeMin/processingMin`):
- `overlap` = блокът застъпва ≥1 `BusyBooking`.
- ако `!overlap` → `free`/`past` (както сега).
- ако `overlap && allowParallel && pairFits` с правилата → `parallel`.
- иначе → `busy`.

`getDaySlots` подава `processingMin` (вече подава `activeMin`).

### 3.4 Канали за запис (`staff-bookings.ts`)

`createMyBooking`, `rescheduleMyBooking`, `editMyBooking` извикват `fitsParallelSlot(...)` (с `excludeId` при move/edit) вместо стария `fitsParallelWindow`. Маркират `allowParallel=true` при паралелен слот (за изключване от EXCLUDE constraint). Snapshot на `activeMin/processingMin` при create (както сега).

### 3.5 Защита от двоен запис

Остава двупластова: PARTIAL EXCLUDE constraint `bookings_no_overlap WHERE (allow_parallel=false)` на ниво база + `fitsParallelSlot` на ниво приложение преди всеки запис. Паралелните (allowParallel=true) се изключват от constraint-а, но винаги минават app-проверката.

## 4. Архитектура — част B (престой в /staff)

**Решение:** престоят остава **глобален** (на `service_items.activeMin/processingMin`), но става **редактируем от /staff**. Обосновка: престоят е физична характеристика на услугата (боята стои Х мин, независимо кой я прави); Снежана е единственият hair изпълнител → глобално = практически нейно; **без рискова prod DB миграция**. `ownOffering` вече чете active/processing от `service_items` — без промяна.

Промени:
- `resource-services.ts` → `updateMyService` schema += `activeMin?`, `processingMin?`; при подадени стойности обновява `service_items.activeMin/processingMin` за съответната услуга (+ `revalidateCatalog`). Own price/duration си остават в `resource_services`.
- `my-services.tsx` → `MyServiceOpt` += `activeMin`, `processingMin`; `EditSheet` += два input-а „Активни минути" / „Минути престой" с кратък помощен текст („за паралелни часове по време на престой; 0 = няма престой"). `AddSheet` — по избор, по-късно.
- `/staff/services/page.tsx` подава active/processing към `MyServiceOpt`.

⚠️ Понеже е глобална промяна от staff, текстът да го подсказва дискретно („за всички, които предлагат услугата"). За практиката (1 hair изпълнител) е безопасно.

## 6. Edge cases (→ тестове)

| Случай | Очаквано |
|---|---|
| 17:30 стои (с престой), добавя 17:00 (reverse) | `parallel` ✅ |
| 17:00 стои, добавя 17:30 (forward, регресия) | `parallel` ✅ |
| 17:00 дълга коса (намазване 40 мин → 17:40) + час 17:30 | намазвания се блъскат → `busy` ⛔ |
| Ново намазване пада върху чуждо измиване (не престой) | правило 2(b) фейлва → `busy` ⛔ |
| Услуга без престой (стрижка / стара „Боядисване" 0/0) | няма прозорец → `busy` ⛔ |
| Move/edit на собствен паралелен час (excludeId) | не се брои сам себе си ✅ |
| Престой прелива след работно време | разрешено (намазването е в деня) ✅ |
