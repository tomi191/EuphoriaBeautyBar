# Дизайн: Паралелни записи (gap/processing booking) в staff PWA

> Дата: 2026-06-24 · Статус: одобрен дизайн, чака spec review
> Цел: изпълнителят да записва втори клиент в „престоя" на боядисване/балаяж — напр. 17:30 боядисване + 18:00 паралелно боядисване.

## 1. Проблем

Боядисването, балаяжът и кичурите имат две части: **активно нанасяне** (стилистът работи) и **престой** (косата стои с боята, стилистът е свободен). Снежана иска да приема втори клиент по време на престоя — стандартна salon практика (double-booking / gap booking).

Системата има основата (`parallel.ts`, `windowFor`, `allowParallel`, полета `activeMin`/`processingMin`), но:
1. **Данни:** цветните услуги са с `active=0, processing=0` → `windowFor` връща NULL → няма паралелен прозорец.
2. **Форма:** „Нов час" (`staff-booking-form` + `fetchMySlots`) не подава `allowParallel` → паралелните слотове излизат `busy` (задраскани), не записваеми.
3. **Логика:** паралелната проверка е по **целия блок** (90 мин), не по **активното** време → две боядисвания никога не се събират едно в друго.

## 2. Решение (overview)

Индустриалният 3-фазен модел (Vagaro/Boulevard: application + development + finish), постигнат **върху съществуващата 2-полева архитектура** чрез конфигурация — без нова schema:

```
durationMin = activeMin (нанасяне) + processingMin (престой/development) + finish (остатък, авто-резервиран)
```

Прозорецът за паралел (`windowFor`) е само в престоя; остатъкът (finish) остава зает в края на блока → измиването е резервирано, точно като индустрията.

### Времева диаграма (17:30 + 18:00, боядисване 25/45/20)
```
17:30 ─ 17:55 ──────── 18:40 ─ 19:00
 A:  нанасям │ престой  │ мия A │
            18:00 ─ 18:25 ──────── 19:05 ─ 19:30
 B:          нанасям │  престой   │ мия B │   (паралелно, в престоя на А)
```
Прозорец на A: `17:30 + (25+5)` … `17:30 + (25+45−5)` = **18:00–18:35**. Активното на B (18:00–18:25) се събира в него. ✅

## 3. Данни — active/processing per цветна услуга

Предложени стойности (Снежана потвърждава/коригира; сумата = текущия `durationMin`, остатъкът е „измиване"):

| Услуга | durationMin | activeMin (нанасяне) | processingMin (престой) | finish (авто) |
|--------|-------------|----------------------|--------------------------|---------------|
| Боядисване | 90 | 25 | 45 | 20 |
| Балаяж | 180 | 45 | 90 | 45 |
| Кичури на фолио | 90 | 30 | 40 | 20 |
| Корекция на цветовете | 120 | 30 | 60 | 30 |

Услуги БЕЗ престой (подстригване, сешоар, маникюр и т.н.) остават `active=0, processing=0` → работят както сега (без паралел).

**Ниво на запис:** `ownOffering` чете `activeMin`/`processingMin` от **`serviceItems` (каталога)**, а не от `resourceServices` (там има само `durationMin`/`bufferMin`). Затова active/processing се задават на **каталожната услуга** — свойство на самата процедура (боядисването винаги се нанася ~25 мин), не на изпълнителя. Стойностите се прилагат със скрипт (DB запис на `serviceItems` — пуска го собственикът, не auto-mode).

## 4. Алгоритъм — active-based паралел

### 4.1 `getDaySlots` (slots.ts)
Добавя опционален `activeMin`. Цикълът става active-based, за да допусне паралелни слотове близо до края на деня:

```
activeMs = (activeMin > 0 ? activeMin : durationMin) * 60000
за t от open, докато t + activeMs <= close, стъпка 15 мин:
  blockEnd  = t + blockMs        // durationMin + buffer
  activeEnd = t + activeMs
  fitsBlock = blockEnd <= close
  overlaps  = застъпва ли busy (по blockEnd)
  ако !overlaps && fitsBlock:        статус = past/free
  иначе ако allowParallel && някой window съдържа [t, activeEnd]: статус = past/parallel
  иначе ако fitsBlock:               статус = busy
  иначе:                             пропусни слота (block overflow без паралел — както сега)
```
Ключово: за паралел се проверява **activeEnd** (не blockEnd) в прозореца → второто боядисване (само активното) се събира, престоят му може да прелее след `close` (решение на собственика).

### 4.2 `fitsParallelWindow` (parallel.ts) — без промяна в сигнатурата
Извикващите (`createMyBooking`, `rescheduleMyBooking`, `editMyBooking`) подават **active end** (`start + activeMin`) вместо block end, така че проверката съвпада с `getDaySlots`. `windowFor`/`parallelWindows` остават както са (вече са active-based за прозореца).

## 5. Компоненти за промяна

1. **`src/lib/booking/slots.ts`** — `getDaySlots` приема `activeMin`, active-based слот логика (4.1).
2. **`src/lib/actions/staff-bookings.ts`**:
   - `fetchMySlots` → подава `allowParallel: true` + `activeMin` към `getDaySlots`.
   - `createMyBooking` → приема `allowParallel` от формата; при `allowParallel` проверява `fitsParallelWindow` с **active end**; записва `allowParallel`, `activeMin`, `processingMin` (вече се пишат).
   - `rescheduleMyBooking` / `editMyBooking` → същата active-end проверка.
3. **`src/components/staff/staff-booking-form.tsx`**:
   - Слот `state` помни и статуса (`free` | `parallel`).
   - Слотове `parallel` стават **clickable** (различен цвят — салвия/mint „в престой", не задраскани).
   - При submit → `createMyBooking({ ..., allowParallel: slotStatus === "parallel" })`.
4. **Данни** — скрипт `scripts/set-service-timing.ts`, който обновява `serviceItems.activeMin`/`processingMin` по таблицата от §3; пуска го собственикът (DB запис).
5. **`booking-edit-sheet`** — вече има полета „Активни минути / Минути престой" за ръчна корекция per час; без промяна (евентуално само уточнение на надписа).

## 6. UX (приоритет: лесно опериране)
- **Автоматично:** изпълнителят записва нормално; паралелните слотове се показват сами в престоите.
- **Визуално разграничими:** паралелните слотове са в различен цвят с етикет (напр. „в престой"), за да е ясно, че е double-booking.
- **Само staff:** публичният `/zapazi-chas` НЕ активира паралел (онлайн клиентите не виждат gap слотовете) — непокътнат.

## 7. Edge cases
- **Overflow след работно време:** позволен за паралел — само активното трябва да е до `close`; престоят/измиването може след (решение на собственика).
- **Bottleneck (две измивания съвпадат):** не се авто-предотвратява — управлява се ръчно (1 стилист, таймер). Приемливо по research; out of scope за тази итерация.
- **Услуга без active/processing:** `activeMin=0` → fallback към `durationMin` → държи се точно както сега (нула регресия).
- **EXCLUDE constraint:** паралелните часове са с `allowParallel=true` → изключени от DB overlap constraint-а (вече имплементирано); валидацията минава през `fitsParallelWindow`.

## 8. Тестване
- Read-only диагностика (`check-coloring.ts`) преди/след — потвърждава, че `windowFor` дава прозорец.
- Визуална репродукция в staff PWA: 17:30 боядисване (host), после нов час 18:00 → трябва да е **записваем паралелен слот** (не задраскан).
- Регресия: кратка услуга (подстригване) и услуга без престой → непроменено поведение.
- Публичен `/zapazi-chas` → без паралелни слотове.

## 9. Out of scope (нарочно)
- Отделна `finish` колона в schema (3-фазната точност се постига чрез конфигурация).
- Bottleneck авто-предотвратяване (мивка/стол) — за бъдеще, при няколко стола.
- Паралел в публичния онлайн booking.
- Автоматично извеждане на active/processing от историята.
