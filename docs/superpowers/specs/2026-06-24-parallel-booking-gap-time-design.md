# Дизайн: Услуги по дължина + паралелни записи (Спринт 1)

> Дата: 2026-06-24 · Статус: одобрен дизайн, чака spec review
> Обхват: **Спринт 1** — цветните услуги × 3 дължини (research-base времена) + gap booking логика (staff) + 3 line-art картинки. Спринт 2 (останалите услуги по дължина) е out of scope тук.

## 1. Проблем

Услугите с химия (**боядисване, балаяж, кичури, корекция на цвят**) имат три фази: **нанасяне** (стилистът работи) → **престой/химия** (косата стои, стилистът е свободен) → **измиване** (стилистът работи). Два проблема:

1. **Не отчитат дължината.** Каталогът пази единична продължителност (боядисване = 90), но research показва, че **нанасянето варира силно по дължина** (къса 15-20 мин, дълга 45-60), докато престоят (химия) е стабилен (~40 мин). Клиентите не могат да запишат точно време.
2. **Няма паралелни записи.** Снежана иска да приема втори клиент в престоя (double-booking). Системата има основата (`parallel.ts`, `windowFor`, `allowParallel`), но услугите са на `0`, staff формата не е свързана, и проверката е block-based вместо active-based.

## 2. Решение (overview)

**(а) Услуги по дължина:** всяка цветна услуга → 3 варианта (къса/средна/дълга) като отделни `serviceItems` (както „Дамско подстригване (къса/дълга коса)" вече). Всеки вариант с research-base `activeMin/processingMin` + своя цена.

**(б) 3-фазен gap booking** върху съществуващата 2-полева архитектура (без нова schema):
```
durationMin = activeMin (нанасяне) + processingMin (престой/химия) + finish (остатък = измиване, авто-резервиран)
```
`windowFor` дава прозорец в престоя; остатъкът (измиване) остава зает в края.

**(в) 3 line-art картинки** (къса/средна/дълга силует, салвия, `mix-blend-multiply`) — показват се при избор на дължина. Преизползваеми навсякъде.

### Времева диаграма (17:30 + 18:00, боядисване средна коса 25/40/20)
```
17:30 ─ 17:55 ──────── 18:35 ─ 18:55
 A:  нанасям │ престой  │ мия A │
            18:00 ─ 18:25 ──────── 19:05 ─ 19:25
 B:          нанасям │  престой   │ мия B │   (паралелно, в престоя на А)
```
Прозорец на A: `17:30 + (25+5)` … `17:30 + (25+40−5)` = **18:00–18:30**. Активното на B (18:00–18:25) се събира в него. ✅

## 5. Алгоритъм — active-based паралел (gap booking)

### 5.1 `getDaySlots` (slots.ts)
Добавя опционален `activeMin`. Цикълът става active-based, за да допусне паралелни слотове близо до края на деня:
```
activeMs = (activeMin > 0 ? activeMin : durationMin) * 60000
за t от open, докато t + activeMs <= close, стъпка 15 мин:
  blockEnd  = t + blockMs        // durationMin + buffer
  activeEnd = t + activeMs
  fitsBlock = blockEnd <= close
  overlaps  = застъпва ли busy (по blockEnd)
  ако !overlaps && fitsBlock:        статус = past/free
  иначе ако allowParallel && някой window съдържа [t, activeEnd]: статус = past/parallel
  иначе ако fitsBlock:               статус = busy
  иначе:                             пропусни слота (block overflow без паралел — както сега)
```
Ключово: за паралел се проверява **activeEnd** (не blockEnd) → второто боядисване (само активното) се събира, престоят му може да прелее след `close`.

### 5.2 `fitsParallelWindow` (parallel.ts) — без промяна в сигнатурата
Извикващите (`createMyBooking`, `rescheduleMyBooking`, `editMyBooking`) подават **active end** (`start + activeMin`) вместо block end. `windowFor`/`parallelWindows` остават както са.

## 7. UX + двата режима

**Публичният booking ВЕЧЕ е свързан с паралел** (`fetchPublicSlots(..., true)` в `public-booking-form.tsx:218` + `allowParallel: slotIsParallel` ред 260). Естествено разграничение:

| | Публичен (онлайн) | Staff (Снежана) |
|---|---|---|
| Логика | **block-based** (вече) | **active-based** (нова) |
| Вмъква в престоя | само **кратки** услуги (цяло в престоя) | и **дълги** (второ боядисване) |
| Код | **никаква промяна** — само активиране с данни | свързване (компонент 4-5) |

- **Staff:** автоматично — паралелните слотове се показват сами, в различен цвят („в престой").
- **Публичен:** block-based значи онлайн клиент вижда паралел само за кратка услуга, която цяло се събира в престоя (безопасно, измиванията не се застъпват). Дълго второ боядисване — само staff.

## 8. Edge cases
- **Overflow след работно време:** позволен за паралел — само активното до `close`; престоят може след.
- **Bottleneck (две измивания съвпадат):** не се авто-предотвратява — ръчно (1 стилист, таймер).
- **Услуга без active/processing:** `activeMin=0` → fallback към `durationMin` → както сега (нула регресия за маникюр/козметика).
- **EXCLUDE constraint:** паралелните часове (`allowParallel=true`) са изключени от DB overlap-а; валидира `fitsParallelWindow`.

## 9. Тестване
- `check-coloring.ts` — потвърждава `windowFor` дава прозорец след конфигурацията.
- Staff PWA: 17:30 боядисване (host), после 18:00 → записваем паралелен слот (не задраскан).
- Регресия: маникюр/подстригване (без престой) → непроменено.
- Публичен: кратка услуга в престоя → показва се; дълго второ боядисване → не се показва.
- Каталог `/uslugi`: length-вариантите групирани коректно; цени per дължина.
